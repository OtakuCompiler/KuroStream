## 1. Why five different UIs don't all "just work" together

Five codebases means five competing ideas of what the home/search/details/player screens
should be, in two different UI toolkits (Jetpack Compose vs. Fragments+XML+ViewBinding),
with at least **four incompatible `MediaItem`-shaped models** (`domain.model.MediaItem`,
`tv.model.MediaItem`, `extensions.domain.model.CatalogItem`/`MediaDetail`,
`legacy.entity.MediaItem`) plus a fifth outside this project's `domain.entity.MediaItem`.
I did not attempt to unify all of these into one model — that's a real product/architecture
decision (which fields does "the" MediaItem need?) that isn't safe to make silently. Instead:

- **`:app` is the one real, live, compiling app UI** (Compose + tv-material, from phases
  31-40). It's wired to the canonical, Room-backed `:data` layer — see section 2.
- **`:extensions`** ships a second, complete set of Compose home/search/details screens
  (from phases 61-70) plus real third-party integrations (TorrServer, Stremio, Kitsu,
  Cloudstream). These are **not** wired into `:app`'s navigation graph. They're real,
  substantial code — worth pulling individual pieces (e.g. the TorrServer screen, the
  Kitsu repository) into `:app` deliberately, rather than running two parallel home
  screens.
- **`reference/anistream-fragment-ui/`** (phases 41-60) is Fragment/XML UI kept purely
  for reference. It cannot compile as-is (see section 4) and duplicates screens `:app`
  already has in Compose.
- **`:launcher`** (phases 81-90) is a genuinely distinct feature (pick a Plex/Jellyfin/
  Emby server on first run) that's plausible as a real second entry point/Activity in
  `:app`, once its missing pieces (section 5) are filled in.

## 2. `:app` — what's actually wired vs. approximated

Phase 31-40's `HomeViewModel`/`DetailsViewModel`/`PlayerViewModel` were written against a
`MediaRepository`/`WatchProgressRepository`/`FavoritesRepository` trio with a different
shape than this project's canonical (phases 11-20) `domain.repository.MediaRepository`.
Rather than let a third incompatible `MediaRepository` collide with the canonical one,
I moved the tv-shaped interfaces to `com.kurostream.tv.repository` and wrote real adapter
implementations (`TvRepositoryAdapters.kt`) that delegate to the canonical, Room-backed
repositories:

- **Genuinely working:** `getMediaById`, favorites (`isFavorite`/`toggleFavorite`),
  continue-watching, save-progress, playback URL lookup. These use real Room-backed data.
- **Approximated, clearly commented in code:** `observeTrending`/`observeNewReleases`
  just query `observeMediaByCategory` (no real "trending" ranking exists in the canonical
  model yet). `observeSeasonal` returns an empty list — there's no "season" concept
  anywhere in this project's domain model.
- **Stand-ins, not real features:** `getEpisodes` returns a single synthetic Episode
  wrapping the MediaItem itself, and `getNextEpisode` always fails — the canonical domain
  model has no per-series episode list at all (each `MediaItem` is one watchable unit).
  Building this properly needs a real `Series`/`Episode` addition to `:domain`.

I also fixed a pre-existing bug while touching `PlayerViewModel.kt`: `getPlaybackUrl`
returns `Result<PlaybackUrl>` (an object with `.url`/`.title`), but the original code
passed the success value directly to `ExoMediaItem.fromUri(...)` as if it were a bare
string. Fixed to use `.url` correctly.

The manifest, theme, and `app_name`/`browse_title` strings were rebranded from
"AnimeStream" to "KuroStream"; internal style/class names (`Theme.AnimeStreamTV`,
`AnimeStreamTvApplication`) were left as-is — purely cosmetic, low value to chase across
every file for the time it'd cost. `android:icon` points at `@mipmap/ic_launcher`, which
doesn't exist in any archive — add a real launcher icon before building.

## 3. Duplicate/competing features across archives (not reconciled into one)

Several features were implemented independently in more than one archive, with no shared
code:

- **Watch party:** `extensions.watchparty` (61-70, plain WebRTC + signaling client) vs.
  `playback.advanced.watchparty` (71-80, WebRTC + CRDT-style sync + LAN discovery via
  mDNS). The 71-80 version looks more complete; I'd recommend standardizing on it and
  deleting the other, but didn't delete anything myself.
- **Community notes:** `extensions.community` (61-70, ViewModel + demo data) vs.
  `playback.advanced.community` (71-80, Firestore-backed `CommunityNotesManager`). Same
  recommendation — 71-80's is the more production-shaped one.
- **ASR captions:** `extensions.asr` (61-70, Android `SpeechRecognizer` + Vosk) vs.
  `playback.advanced.captions` (71-80, Whisper tokenizer + mel-spectrogram pipeline —
  a materially different, on-device Whisper-style approach vs. Vosk/Android ASR).
- **Sync providers:** canonical `data.sync.DemoCloudSyncProvider` (implements
  `domain.sync.SyncProvider`) vs. `data.anistream.sync.{AnilistSyncProvider,
  MalSyncProvider, FirestoreSyncProvider}` (implement their own, differently-shaped
  `data.anistream.sync.SyncProvider`). The anistream ones are real, concrete
  implementations against real APIs (AniList GraphQL, MAL, Firestore) — worth adapting
  onto the canonical `domain.sync.SyncProvider` interface as a follow-up.
- **ML runtimes:** three different ones are now dependencies of this one project —
  PyTorch Mobile (`playback.advanced.ai`, super-resolution/frame-interpolation),
  TensorFlow Lite (`launcher.ml.prediction`, next-episode prediction), and none needed
  for Vosk (has its own runtime). Consider consolidating if binary size matters.
- **JSON libraries:** Gson (canonical `:data`), Moshi (`:extensions`), kotlinx.serialization
  (declared in the catalog from phases 1-30 but seemingly unused now) are all
  dependencies simultaneously. Harmless but worth consolidating.
- **Video player engines:** `:playback` uses Media3 ExoPlayer + libVLC + MPV;
  `:launcher` uses **legacy `com.google.android.exoplayer2.*`** (pre-Media3). These are
  different artifacts with overlapping class names in some cases — if both modules end up
  in the same APK, watch for duplicate-class / resource-merge errors. Recommend porting
  `:launcher`'s player usage to Media3 (the API surface is very similar) rather than
  shipping both.

## 4. `reference/anistream-fragment-ui/` (phases 41-60 UI)

This is genuinely non-buildable as delivered — the archive **shipped zero layout XMLs
and no `build.gradle`** for the Fragment UI, so every `R.layout.fragment_xxx` reference
is dangling regardless of anything I did. Kept as read-only reference material, not part
of any Gradle module. Its **data layer** (`AddonRepository`, `DownloadManager`,
`SearchRepository`, `SettingsRepository`, `SubtitleProvider`/`SubtitleMatcher`/
`SubtitleCache`, `BackupManager`, `IntroSkip*`, three real sync providers) had no such
problem and was promoted into `:data` as `com.kurostream.data.anistream.*` — it's real,
substantial, working business logic. Needs Firebase, Ktor (a different HTTP client than
the rest of `:data`'s Retrofit), and AppAuth added as dependencies (done).

## 5. `:launcher` (streambox, phases 81-90) — real data-quality problems in the source zip

This is not a merge artifact — I'm flagging it because it'll be confusing otherwise.
**32 of the 83 files in `streambox_phases_81_90.zip` were shipped completely empty
(0 bytes)**, specifically:

- All 4 Hilt DI modules (`CacheModule`, `ExtensionsModule`, `FirebaseModule`, `MLModule`)
- Most of the Firebase layer: `AuthState`, `AuthResult`, all of `remoteconfig/*`, all of
  `messaging/*` (FCM service, notification helper, Crashlytics, performance monitor)
- `ml/prediction/PredictionModel.kt`
- **Every single resource file**: all 5 layout XMLs, both `values/*.xml` files, the
  preferences XML
- **Both ML assets**: `next_episode_model.tflite` and `prediction_labels.txt` (0 bytes —
  there is no actual trained model in this archive)

I recreated the empty `.kt` files as explicit stub files with a `// MERGE NOTE` comment
so the class names at least resolve (real files like `FirebaseAuthManager.kt` and the
three `*ExtensionsModule.kt` files reference these types and would otherwise fail to
compile with "unresolved reference" errors with no explanation). **I did not invent
implementations for any of them** — that would mean guessing at real auth/DI/ML logic.
The layouts and ML assets are copied through as empty files too, for the same reason
(so the resource IDs exist) — replace with real content before shipping.

The four `.github/workflows/*.yml` CI files in that archive did have real content, but
reference build paths that don't match this merged project's structure — left out of
the zip rather than included-but-stale; recreate them against the new module layout if
you want CI back.

## 6. Mechanical fixes (same spirit as the phases 1-30 report)

- Package renames verified with a full-tree grep after each pass — no leftover
  `com.animestream`/`com.anistreamtv`/`com.anime.stream`/`com.streampulse`/`com.streambox`
  references anywhere except my own explanatory comments.
- Fixed **JNI symbol names** in `MpvRendererJNI.cpp/.h` and `native_dsp.cpp` to match the
  renamed Kotlin packages (`Java_com_streampulse_player_...` →
  `Java_com_kurostream_playback_advanced_...`) — these have to match exactly or the
  native library won't bind to its Kotlin caller at runtime.
- Fixed matching manifest permission/action strings in the phase 76 sandbox service
  (`com.streampulse.permission.RUN_EXTENSION` → `com.kurostream.permission.RUN_EXTENSION`,
  etc.) so the manifest and the Kotlin code that reads these strings still agree.
- Wrote a combined `playback/src/main/cpp/CMakeLists.txt` building both native targets
  (`mpvrenderer` from phase 71, `kurostream_dsp` from phase 80) and re-enabled
  `externalNativeBuild` in `playback/build.gradle.kts` (previously disabled in the phases
  1-30 merge because no native sources existed yet — they do now). Both native targets
  still need their upstream libraries resolvable: libmpv-for-Android (same manual step as

  Moshi, Vosk, WebRTC, NanoHTTPD, PyTorch Mobile, Oboe, legacy ExoPlayer2, TensorFlow

  `ProfileRepository`, `MediaItem`, `HomeScreen`, etc. — 14 in total) was checked and
  confirmed to sit in a distinct package. Full list is in the merge working notes if
  you want it; none require action.
