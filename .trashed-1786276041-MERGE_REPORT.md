# KuroStream — Merge & Reconciliation Report

This merges three archives into one project, following the merge instructions in
`phase_11_20`'s and `phase_21_30`'s README files:

- `KuroStream_AllPhases_1-10.zip` (app skeleton, common, domain, plugin-sdk)
- `kurostream-phases-11-20.zip` (data, cache modules — README included)
- `playback_module_phases_21_30.zip` (playback module — README included)

The READMEs' instructions were followed where accurate. Several places where the
READMEs' assumptions didn't match the actual code were found and had to be resolved
by hand — documented below so you can double check my judgment calls.

## 1. Genuine conflicts (not mechanical — required a judgment call)

### `domain/repository/MediaRepository.kt` and `ProfileRepository.kt`
Both phase-1-10 and phase-11-20 define **different, incompatible** interfaces at the
exact same file path/package. The phase-11-20 README says the phase-1-10 versions
should be kept and the new ones "skipped" — but that's backwards: the actual
`MediaRepositoryImpl`/`ProfileRepositoryImpl` in `:data` implement the **phase-11-20**
signatures (`observeMediaByCategory`, `saveWatchHistory`, PIN-based profile switching,
etc.), against `domain.model.*` types, not the phase-1-10 `domain.entity.*` types.

**Decision:** kept the phase-11-20 versions as the canonical
`domain/repository/{MediaRepository,ProfileRepository}.kt`, since they have real
backing implementations. Moved the phase-1-10 versions to
`domain/legacy/repository/` (package `com.kurostream.domain.legacy.repository`) so
nothing is deleted.

### `domain/usecase/*` (GetTrending, GetHomeRows, GetContinueWatching, GetAnimeDetails, SearchAnime)
These phase-1-10 use cases are written against the old `MediaRepository`/`ProfileRepository`
interfaces and the `domain.entity.*` model. Since those interfaces moved to `legacy`,
the use cases moved with them, to `domain/legacy/usecase/` (package
`com.kurostream.domain.legacy.usecase`), with imports repointed at the legacy interfaces.
**They will not compile against `:data`'s implementations as-is** — there's no
`MediaRepositoryImpl` for the old interface. Treat these as reference/scaffolding to
rewrite against the new `domain.model` types, not working code.

### Two incompatible `Profile` classes
- `domain.entity.Profile` (phase 1-10): `displayName`, `isPremium`, `preferredQuality`, etc.
- `domain.model.Profile` (phase 11-20): `name`, `hasPin`, `isActive`, `preferencesJson`.

These live in different packages so there's no file collision, but anyone editing
this code needs to know **`domain.model.Profile` is the one backed by Room** (see
`ProfileEntity.kt` / `ProfileRepositoryImpl.kt`); `domain.entity.Profile` is only
used by the legacy use cases above.

### Two `SyncProvider` interfaces
- `domain.repository.SyncProvider` (phase 1-10, entity-based) → moved to
  `domain.legacy.repository.SyncProvider`.
- `domain.sync.SyncProvider` (phase 11-20) → kept as canonical; implemented by
  `DemoCloudSyncProvider` in `:data`.

These are in different packages so there was no hard collision, only naming confusion.

### Two `Result` conventions
Phase 1-10 code (`common`, legacy use cases) uses a custom sealed class
`com.kurostream.common.result.Result` (`Loading`/`Success`/`Error`). Phase 11-20's
`:data` implementations use Kotlin's stdlib `kotlin.Result` (`success`/`failure`).
Both compile fine independently, but it's worth picking one convention going forward.

### Package namespace mismatch: playback module
The phase 21-30 archive was generated under `com.mediaplayer.playback.*`, not
`com.kurostream.*` like the rest of the project (and its `build.gradle`'s
`namespace` matched). **Renamed** every package declaration/import and the Gradle
`namespace` to `com.kurostream.playback.*` so it matches the rest of the codebase.
Files were also placed into their documented subpackage folders
(`core/`, `selector/`, `mpv/`, `vlc/`, `media3/`, `buffering/`, `display/`,
`diagnostics/`) per the phase 21-30 README's tree — the zip had them flat.

Note the playback module has **no dependency on `:domain`'s `MediaItem`/`PlaybackState`**
— it defines its own `MediaItem`, `PlaybackState`, etc. in `playback.core`. These are
unrelated to `domain.entity.MediaItem`/`domain.model.MediaItem`. A future phase will
need a mapper between them; none was provided.

## 2. Mechanical gaps filled in (not conflicts, just missing pieces)

- **`data/build.gradle.kts` was missing entirely** from the phase 11-20 zip (the
  README documents it as a "Phase 11" deliverable). Reconstructed from the module's
  actual imports (Room, DataStore, Retrofit+Gson, WorkManager+Hilt) and the sibling
  `cache/build.gradle.kts` for style consistency.
- **`cache/build.gradle.kts` referenced catalog aliases that don't exist**
  (`libs.coroutines.android`, `libs.espresso`, `libs.versions.compileSdk/minSdk`, a
  bare-string `disklrucache` dependency instead of a catalog entry). Fixed to use
  real/added aliases.
- **`gradle/libs.versions.toml`** — added: `compileSdk`/`minSdk` version entries,
  `gson`, `retrofit-converter-gson`, `androidx-hilt-work`/`androidx-hilt-compiler`,
  `disklrucache`, `timber`, `media3-*` (8 artifacts), `libvlc-all`, `mpv-android`.
  (Did **not** add the Firebase entries the phase 11-20 README mentions — no code in
  any archive actually imports Firebase, so they'd be dead dependencies.)
- **`settings.gradle.kts`** already included `:data`, `:cache`, `:playback` — the
  READMEs' "add these to settings.gradle" steps were already done in phase 1-10's file.
  However it also includes `:downloader` and `:sync`, which **don't exist in any of
  the three archives** — presumably later phases. Commented these two out so Gradle
  sync doesn't fail; re-enable once those modules exist.
- **`playback/build.gradle.kts`** was Groovy (`build.gradle`) while every other
  module uses Kotlin DSL — converted to `.kts` for consistency. Also **removed the
  `externalNativeBuild`/CMake/NDK `abiFilters` block**: it points at
  `src/main/cpp/CMakeLists.txt`, which isn't in the archive, so as shipped it would
  fail Gradle configuration outright. The MPV dependency is commented out for the
  same reason — the README's "Native Libraries Setup" step (building the mpv-android
  AAR by hand) is still a real manual prerequisite.
- **Missing `proguard-rules.pro`/`consumer-rules.pro`** placeholders created for
  `cache`, `data`, `playback` (referenced by their build files but absent from the
  zips). The playback module's proguard rules block from its README was moved into
  the module's own `proguard-rules.pro`, and also merged into `app/proguard-rules.pro`
  as the README instructs (with the package name corrected).
- **`app/build.gradle.kts` and `app/src/` were entirely absent** from the phase 1-10
  archive (only `app/proguard-rules.pro` shipped, despite the README describing it as
  "Phase 1-6 skeleton"). Added a minimal stub that wires in every module
  (`:common`, `:domain`, `:data`, `:cache`, `:plugin-sdk`, `:playback`) plus a
  placeholder `AndroidManifest.xml`, so the project is at least structurally
  complete. **No Activities, Compose screens, or `@HiltAndroidApp` Application class
  were supplied anywhere in the three archives** — those still need to be written.

## 3. What compiles cleanly as-is

- `:common`, `:plugin-sdk` — unchanged, no conflicts.
- `:cache` — unchanged source, gradle file patched.
- `:data` — unchanged source; new gradle file; depends cleanly on the canonical
  (phase 11-20) domain interfaces.
- `:domain` — canonical model (`domain.model.*`, `domain.repository.*`,
  `domain.sync.SyncProvider`) is consistent and matches `:data`'s implementations.
  `domain.entity.*` and `domain.legacy.*` remain for reference but are orphaned
  (nothing implements or calls them except each other).
- `:playback` — internally consistent after the package rename; standalone (only
  depends on `:domain` in the build file, but doesn't actually import anything from
  it yet).

## 4. Suggested next steps

1. Decide whether to delete `domain/legacy/**` outright or actually port the old
   use cases onto `domain.model`/the new repositories.
2. Write a mapper between `playback.core.MediaItem`/`PlaybackState` and
   `domain.model.MediaItem` so the playback module can actually be wired to
   `:data`'s `MediaRepository`, per the phase 21-30 README's "Integration Points" table.
3. Supply the actual `:app` module (Activities/Compose UI, `AndroidManifest.xml`
   with a launcher activity, `@HiltAndroidApp` class) — none of that existed in any
   of the three uploads.
4. Follow the phase 21-30 README's "Native Libraries Setup" step to produce the
   MPV AAR, then re-enable `externalNativeBuild`/`mpv-android` in
   `playback/build.gradle.kts`.
